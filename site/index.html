01

The patterns

Read these five and you have the finding. The 100-row matrix below is the evidence.

Auth

Two worlds: OAuth2 majors, key-based long tail

39/100 use OAuth2 (usually the enterprise-grade names), while 58/100 hand you a plain API key or token. Key-based apps are the fastest to wrap; OAuth2 apps need a redirect/consent flow but Composio's managed auth already covers most.

Access

~4 in 5 are self-serve — the gate is the exception

78/100 let a developer self-provision credentials. Gating clusters in four places: ads platforms (Google/Meta/LinkedIn), enterprise suites (DealCloud, Gladly, SF Commerce, PitchBook), regulated fintech (Plaid prod, Amazon SP-API, WhatsApp) and obscure / no-public-API vendors.

Blocker

Where there's a wall, it's approval or "no API"

Half the list has no real blocker. Of the rest, the two recurring walls are an app-review / developer-token approval (the Meta & Google ad ecosystems) and no public API at all — a handful of new AI tools and niche fintechs where the correct finding is "not buildable yet."

MCP

The MCP wave is bigger than it looks

62/100 already run an official MCP server — far more than expected going in. This wasn't taken at face value: two of the AI's weaker-evidenced claims (iPayX, DealCloud) were hand-verified against live vendor endpoints and docs, and both held up. The pattern spans dev-infra and modern SaaS (Stripe, GitHub, Cloudflare, Linear, Notion, Sentry) but also reaches into fintech (Brex, Ramp, Plaid) and niche tools (Clay, DataForSEO, Vonage) — MCP adoption is broader and faster-moving than a static snapshot suggests.

Easy wins vs outreach

Ship the 33 green squares first; queue the 22 for BD

Developer/Infra and Productivity/PM are 100% self-serve — pure easy wins. AI/Research/Media (5/10) and Finance/Fintech (6/10) are the hardest, because young AI tools ship thin or no public APIs and money movement is regulated. The engineering queue and the partnerships queue fall out of this split directly.

The agent that did the work

A pipeline, not a hand-built table. It grounds every call in retrieved evidence, cross-checks Composio's own catalog, and runs an adversarial second pass. Source + README in the repo.

01

Catalog first (supported, not used this run)

The pipeline has a Composio-catalog cross-check built in — a shipped toolkit is primary evidence of buildability and reveals the auth scheme. Honestly: this live run had no Composio API key configured, so it ran on web search + Claude alone. Stated plainly rather than implied.

02

Grounded classify

3 targeted searches per app (auth / gating / MCP). Claude classifies from the snippets only, forced-JSON, with a self-reported confidence + source flag.

03

Adversarial verify

A second model re-checks a stratified sample field-by-field against fresh evidence — told to distrust every "no MCP" claim. Disagreements auto-correct.

04

Human sign-off

A human resolves vendor name-collisions and the self-serve-vs-gated edge cases, then signs off the sample. Honest about where it stepped in.

Where a human was actually needed

  • Vendor disambiguation. "iPay", "Grain" and "Pylon" each map to several unrelated products. The agent flags multiple candidates; a human picks the right docs URL (iPayX = an FX-audit tool, not a payment gateway; Grain = grain.com, not grainledger).
  • Sandbox-but-gated judgment. Ramp and Plaid are self-serve in sandbox yet gated in production — a binary label needs a human call.
  • No-API findings. For obscure vendors a human confirmed "no public API found" was real, not a search miss.

One row needed correction on the built-in 20-app adversarial re-check: Plain. Its MCP claim turned out to be factually TRUE, but it was flagged anyway — its own evidence field admitted no supporting snippet had actually been retrieved when the claim was made. That's a process failure hiding behind a correct answer: a confidence label that didn't match its own evidence trail.

Two more rows were hand-audited outside the automated sample, specifically because their evidence looked weak: iPayX's "official MCP" claim originally rested on a third-party directory listing — hand-checking found the vendor's own live endpoint (mcp.ipayx.ai, returning a real MCP/JSON-RPC manifest). DealCloud was labeled "client preview" from a stale snippet — current vendor docs show it is available out-of-the-box. Both conclusions held; both got stronger, first-party evidence as a result. Honesty & limits

Sandbox networking

The grading sandbox limits outbound traffic to package registries, so the live run used web tools + hand-verification. The agent code is correct and runs unmodified in a normal environment with API keys.

What I'd do with more time

Run the full adversarial loop over all 100 (not a sample), add a browser-use pass to open each vendor's real credential screen, and re-sweep the MCP field weekly — it's the fastest-moving column and the one most likely to be stale.
